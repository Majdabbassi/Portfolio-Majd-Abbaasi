Vision & Product Scope

Not a simple ordering app

CaféResto models a complete restaurant environment where clients order via QR code, staff manage orders and table flow, owners configure operational rules, and stock updates automatically based on recipes. The goal was to design a configurable, scalable domain architecture capable of adapting to different restaurant behaviors.

Architecture Strategy

Multi-Tenant Model

Single PostgreSQL database shared across all restaurants with strict data isolation using ownership-based filtering (cafeResto_id). Authorization goes beyond roles — every resource is validated against ownership.

Single PostgreSQL DB

Shared database with tenant isolation by foreign key ownership chains.

Ownership Validation

Authorization beyond roles — every resource verified against restaurant ownership.

Flyway Migrations

Controlled forward/rollback migrations. Hibernate validates only — no auto schema generation.

Production Discipline

ddl-auto=validate enforcing schema discipline in production environments.

Transactional Integrity

Engineering

What Was Hard

The most complex problem was not feature implementation — it was architectural modularization. Designing a domain capable of toggling entire subsystems while preserving transactional integrity and code clarity.

01

Modular Subsystems

Enabling and disabling features (stock, shifts, reservations) without breaking core order flow.

02

No Spaghetti Branching

Keeping conditional logic clean and centralized within the service layer, avoiding scattered if-else chains.

03

Transaction Safety

Ensuring rollback consistency across multi-aggregate operations when any validation fails.

04

Feature Flag Clarity

Maintaining readable, maintainable code while supporting dynamic behavior changes per restaurant.

Reflection

Lessons & Future Improvements

What I'd improve

  • Implement ingredient reservation for concurrency safety
  • Introduce domain events for cleaner module decoupling
  • Add CI/CD automation pipeline
  • Improve structured logging & tracing
  • Consider splitting heavy subsystems into microservices if scale demands

What I learned

  • Multi-tenant isolation requires discipline at every query level
  • Feature-gated architecture demands clean service boundaries
  • Production monitoring is not optional — it's foundational
  • Docker Compose can power real production single-server deployments
  • Schema control (Flyway) prevents silent production breakage
